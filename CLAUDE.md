# CLAUDE.md — 项目编码风格

本文件由 Claude Code 在每次会话开始时自动读取。
用户可按偏好直接修改。规则越具体，AI 生成的代码越贴合预期。

---

## 1. 总原则

- **简洁优于完备**：能用 10 行表达清楚的事，不要写 30 行
- **直白优于聪明**：避免炫技、巧妙的元编程、隐式约定
- **失败要响亮**：错误自然抛出，不要 `try/except` 兜底掩盖
- **不要装饰**：emoji / 装饰符号（↪ ✓ ❌ ✅）一律不用
- **不要播报过程**：删除 `"正在执行 X..."` 类的状态打印；输出只在结果产生时给

---

## 2. Python 风格

### 类型注解

- 优先 **PEP 604 联合类型**：`str | None`，不写 `Optional[str]`
- 优先内置泛型：`list[str]`，不写 `List[str]`
- 必须 `from __future__ import annotations` 放在每个新 `.py` 文件顶部
- 函数签名必须有完整类型注解，包括返回值

### 命名

- 模块/文件：`snake_case`
- 函数/变量：`snake_case`
- 类：`PascalCase`
- 常量：`UPPER_SNAKE`
- 私有：单下划线 `_helper`，不用双下划线（避免 name mangling）

### 数据结构选择

- **TypedDict**：用于 LangGraph state（必须）
- **dataclass**：纯数据容器，无校验需求
- **Pydantic BaseModel**：需要校验 / 序列化（如 LLM 输出解析）
- 避免 `namedtuple` 和裸 dict 当结构体

### 错误处理

- **默认不写 `try/except`**。让异常自然向上传播
- 仅在以下场景允许 catch：
  1. 真的有恢复策略（如重试、降级）
  2. 在最外层入口（`main()` / CLI）转换为用户可读消息
- 永远不写 `except Exception: pass`
- 不写"防御性"参数校验，类型注解 + 调用方负责

### Import 顺序（ruff 会自动排）

1. 标准库
2. 第三方库
3. 本项目（`from cca.xxx import ...`）

## 3. 注释与文档

### docstring

- **越短越好**：一行 docstring 就够，复杂的再写多行
- 写"做什么"，不写"怎么做"
- 不需要描述参数类型（类型注解已说明）
- 示例：

```python
# 好
def deepseek_bullets(state: HelloState) -> dict:
    """列 3 个核心要点。"""

# 不好
def deepseek_bullets(state: HelloState) -> dict:
    """
    This function takes the state, extracts the competitor name,
    calls DeepSeek API to generate bullet points...
    Args: state - the current graph state
    Returns: dict with bullets field updated
    """
```

### 行内注释

- **中文 OK**，但只在以下时候写：
  1. 解释"为什么"（非显而易见的设计）
  2. 标 TODO / FIXME
- 不要把变量名翻译成中文当注释（`x = 1  # 设置 x 为 1`）

### 文件头

- 不要 `# -*- coding: utf-8 -*-`（Python 3 默认 UTF-8）
- 不要作者/日期/版权头
- 模块级 docstring 简短说明本文件用途即可

## 4. LangGraph / Agent 项目特定

### Node 函数签名（统一约定）

```python
def my_node(state: SomeState) -> dict:
    """节点用途一句话。"""
    # ...
    return {"field_a": ..., "field_b": ...}  # 只返回要更新的字段
```

- 输入永远是 `state: SomeState`
- 返回 `dict`，只含本节点写的字段（LangGraph 自动 merge）
- 不要返回完整 state

### LLM 调用

- 优先用 `ChatOpenAI`（DeepSeek 走 OpenAI 兼容协议，免装 `langchain-deepseek`）
- 一定要传 `timeout`，不允许无超时调用
- 不要在节点函数里实例化 LLM 客户端；用模块级常量或 `llm/factory.py`

### State / Reducer

- 多节点向同一 list 写入时，用 `Annotated[list[T], add]` 让其累加
- 不要在节点里手动 `state["xxx"].append(...)`，永远返回 dict

### prompts

- system + user prompt 写在 `src/cca/prompts/*.md`
- 不允许 LLM 调用处硬编码长 prompt
- 短指令（< 80 字）可以内联

### 日志 / 审计

- 关键节点的输入/输出写到 state 的 `audit_log` 字段（已约定的 reducer list）
- 不在节点里 `print()` 业务信息（前端订阅 stream 看）
- 调试用 `loguru` 的 `logger.debug()`，不用 `print`

---

## 5. 测试

- 每写一个 agent / skill / tool，**同日**补 `tests/test_xxx.py`
- 单元测试**绝不**调真 API；用 `tests/conftest.py` 里的 `FakeLLM` fixture
- 测试名直白：`test_collector_returns_sources_for_known_competitor`
- 一个测试只 assert 一件事，宁愿多写几个测试

---

## 6. CLI / 脚本

- `scripts/` 下的脚本必须能直接 `python scripts/xxx.py` 跑通
- 入口必须有 `if __name__ == "__main__": main()`
- 命令行打印保持极简，结果优先，过程不报

---

## 7. 提交信息

- commit message 用**英文**
- 标题 < 70 字符，imperative mood：`Add ...` / `Fix ...` / `Refactor ...`
- 复杂改动写 body，列要点
- 不要在 commit 里加 emoji
- 用 Co-Authored-By 标注 AI 协作

---

## 8. 不要做的事（黑名单）

- 不要写 `print("正在执行 ...")` 这种过程播报
- 不要包大段 `try/except Exception`
- 不要无理由的 `@dataclass(frozen=True)` 等装饰器堆叠
- 不要在 production 代码里留 `print` 调试语句
- 不要写 `# type: ignore` 除非有具体原因（注释里说明）
- 不要写自动生成的代码注释（如 `# generated by xxx`）
- 不要起单字母变量名（除 `i / j / k` 等循环计数）
